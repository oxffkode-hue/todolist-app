# TodoListApp 도메인 정의서 - 품질 검토 보고서

- 평가일: 2026-05-12
- 평가 대상: `docs/1-domain-definition.md` (버전 1.0.0)
- 평가자: Business Analyst (Domain Review)
- 평가 방법: 5개 기준 항목별 1~5점 척도 평가

---

## 1. 평가 개요

본 검토는 TodoListApp 도메인 정의서 버전 1.0.0을 대상으로, 소프트웨어 개발 착수 전 도메인 문서의 품질을 5가지 기준(완전성, 명확성, 추적성, 검증 가능성, 유지보수성)으로 평가한다. 평가 결과는 후속 설계 단계(API 설계, ERD, 기능 명세서)의 기준 문서로 활용된다.

---

## 2. 기준별 상세 평가

---

### 기준 1. 완전성 (Completeness)

**점수: 4 / 5**

#### 잘된 점 (Strength)

- 프로젝트의 핵심 세 도메인(User, Todo, Category)을 빠짐없이 식별하고 각각의 속성을 속성명, 타입, 설명 형태로 체계적으로 정의했다.
- 범위(Scope)와 범위 제외(Out of Scope)를 명시적으로 분리하여 협업 시 기능 경계 논쟁을 사전에 차단하는 구조를 갖추었다.
- 비즈니스 규칙(BR-01 ~ BR-13)이 도메인별로 그룹화되어 있고, 필터링 동작 규칙까지 포함하는 폭넓은 커버리지를 보인다.
- 유스케이스 8개가 범위 내 모든 주요 기능(인증, CRUD, 필터링, 카테고리 관리)을 포괄한다.

#### 개선점 (Weakness)

- **인증 토큰 관련 정의 누락**: UC-02에서 "인증 토큰 발급"이 언급되지만, 토큰의 종류(JWT, Session 등), 유효기간, 갱신(Refresh) 메커니즘에 대한 정의가 없다. 이는 보안 설계 및 API 명세 작성 시 혼란을 유발할 수 있다.
- **비밀번호 정책 미정의**: BR-02에서 암호화 저장을 명시하지만, 비밀번호 최소 길이, 복잡도 요구사항 등 입력 유효성 기준이 없다. UC-01, UC-03의 "유효성 검증" 단계의 실질적 기준이 불분명하다.
- **Title 길이 제약 미정의**: BR-04에서 title이 필수임은 명시하나, 최대 길이 등의 제약 조건이 없어 DB 스키마 설계 및 UI 입력 제어 기준이 모호하다.
- **카테고리 updatedAt 속성 누락**: User와 Todo 모델에는 updatedAt이 존재하지만 Category 모델에는 없다. 사용자 정의 카테고리는 수정 기능(BR-09)이 존재하므로 updatedAt 추적이 필요하다.
- **에러 응답 정의 부재**: 유스케이스의 예외(Exception) 흐름이 "거부"라는 단어로만 서술되어 있으며, 구체적인 오류 코드나 응답 메시지 형식이 정의되어 있지 않다.

---

### 기준 2. 명확성 (Clarity)

**점수: 4 / 5**

#### 잘된 점 (Strength)

- 섹션 2의 용어 정의 테이블이 핵심 도메인 개념 8개를 한국어-영어 병기로 명확하게 정의하여 이해관계자 간 용어 혼용을 방지한다.
- 비즈니스 규칙이 "BR-XX" 고유 번호 체계로 관리되어 커뮤니케이션 시 참조가 명확하다.
- 엔티티 관계가 ERD 표기법(Crow's Foot Notation)으로 간결하게 표현되어 있다.
- 기본 카테고리와 사용자 정의 카테고리의 구분 기준(`isDefault`, `userId = null`)이 모델 수준에서 명확하게 표현된다.

#### 개선점 (Weakness)

- **BR-10의 범위 모호성**: "동일 사용자 범위 내에서 카테고리 이름 중복 불허"는 기본 카테고리와 사용자 정의 카테고리 간 이름 충돌 허용 여부를 명시하지 않는다. 예를 들어 기본 카테고리 "업무"와 동일한 이름의 커스텀 카테고리 생성 가능 여부가 불명확하다.
- **BR-11의 처리 방식 미정의**: "삭제할 수 없다"는 정책만 있고, 대안 안내(예: 할일을 먼저 삭제하거나 다른 카테고리로 이동 후 삭제 가능)에 대한 기술이 없어 UX 설계 시 해석 여지가 생긴다.
- **UC-03 "등" 표현**: 개인정보 수정 흐름에서 "이름, 비밀번호 등"의 "등"이 수정 가능한 속성의 완전한 목록을 숨기고 있다. User 모델 기준으로 수정 가능한 필드를 명시적으로 열거해야 한다.
- **완료 상태 전환 규칙 미정의**: 완료 처리 후 미완료로 되돌리는 것(toggle)이 허용되는지 여부가 비즈니스 규칙과 유스케이스 어디에도 명시되어 있지 않다.

---

### 기준 3. 추적성 (Traceability)

**점수: 3 / 5**

#### 잘된 점 (Strength)

- 비즈니스 규칙(BR)과 유스케이스(UC)가 독립적인 번호 체계를 가지고 있어 추후 기능 명세서나 테스트 케이스 작성 시 참조 기반이 형성되어 있다.
- 유스케이스의 전제조건(Pre-condition)이 명시되어 있어 비즈니스 규칙(특히 BR-03, BR-06)과의 개념적 연계가 가능하다.
- 도메인 모델 속성이 용어 정의 테이블과 일관된 명칭을 사용하여 개념-모델 간 연결이 자연스럽다.

#### 개선점 (Weakness)

- **BR-UC 간 명시적 연결 없음**: 각 유스케이스가 어떤 비즈니스 규칙을 적용하는지 문서 내에서 명시적으로 연결되어 있지 않다. 예를 들어 UC-04(할일 등록)는 BR-04, BR-05를 적용해야 하지만, 유스케이스 기술문에 이 참조가 없다. 이로 인해 개발자가 규칙 누락 없이 구현했는지 검증하기 어렵다.
- **요구사항 ID 체계 부재**: 문서 전체에 "REQ-XX" 수준의 상위 요구사항 식별자가 없다. 프로젝트 규모가 커질 경우 비즈니스 규칙과 유스케이스를 상위 요구사항에 연결하는 계층적 추적성이 불가능하다.
- **도메인 모델과 유스케이스 간 매핑 없음**: 각 유스케이스가 어떤 엔티티를 생성/읽기/수정/삭제(CRUD)하는지에 대한 매핑이 없다. 이는 API 설계 단계에서 엔드포인트-엔티티 관계를 재도출해야 하는 비효율을 만든다.
- **변경 이력 관리 미흡**: 버전 1.0.0으로 표기되어 있으나, 변경 이력 테이블(Change Log)이 없어 향후 문서 개정 시 무엇이 변경되었는지 추적이 불가능하다.

---

### 기준 4. 검증 가능성 (Verifiability)

**점수: 3 / 5**

#### 잘된 점 (Strength)

- 비즈니스 규칙 대부분이 "허용하지 않는다", "할 수 없다", "~이어야 한다" 형태의 단언문(assertion)으로 기술되어 있어 Pass/Fail 판정이 가능한 구조를 가진다.
- BR-01(이메일 고유성), BR-05(카테고리 필수), BR-11(할일 존재 카테고리 삭제 불가) 등은 경계값 및 예외 케이스 테스트를 직접 도출할 수 있는 수준으로 기술되어 있다.
- 유스케이스에 예외 흐름이 포함되어 있어 기본 경로(Happy Path) 외 오류 경로 테스트 케이스 도출 근거가 존재한다.

#### 개선점 (Weakness)

- **정량적 기준값 부재**: 비밀번호 복잡도, title 최대 길이, 카테고리 이름 최대 길이, 한 사용자당 최대 할일 등록 수 등 수치 기반의 검증 가능한 기준이 전혀 없다. 이로 인해 경계값 분석(Boundary Value Analysis) 테스트 케이스를 도출할 수 없다.
- **수락 기준(Acceptance Criteria) 미정의**: 유스케이스가 "흐름" 중심으로만 기술되어 있고, 각 유스케이스의 성공 조건(Given-When-Then 형식의 수락 기준)이 없다. QA팀이 독립적으로 테스트 케이스를 작성하기 어렵다.
- **비기능 요건 완전 부재**: 성능(예: 할일 목록 조회 응답시간 2초 이내), 가용성, 보안 강도(예: bcrypt rounds) 등의 비기능 요건이 없어 비기능 테스트의 합격 기준을 정의할 수 없다.
- **BR-12 복합 필터 동작 정의 불완전**: "단독 또는 복합 조건"으로 필터링 가능하다고 했지만, 복합 조건 적용 시 AND/OR 연산자 중 어떤 것이 기본 논리인지 명시되어 있지 않아 구현 결과의 정확성을 검증하는 기준이 모호하다.

---

### 기준 5. 유지보수성 (Maintainability)

**점수: 3 / 5**

#### 잘된 점 (Strength)

- 문서가 5개의 논리적 섹션(개요, 용어, 모델, 규칙, 유스케이스)으로 명확히 분리되어 있어 특정 영역의 변경이 다른 영역에 미치는 영향을 쉽게 파악할 수 있다.
- 비즈니스 규칙과 유스케이스가 독립적인 번호 체계를 가지므로 항목 추가 시 기존 번호 체계를 유지하면서 확장이 가능하다.
- 마크다운 형식으로 작성되어 버전 관리 시스템(Git)에서 diff 추적이 용이하다.

#### 개선점 (Weakness)

- **변경 이력(Change Log) 섹션 부재**: 버전 1.0.0 이후 규칙이나 모델이 변경될 때 무엇이 왜 변경되었는지 기록할 테이블이 없다. 문서가 살아있는 산출물로 관리되려면 변경 이력 관리가 필수이다.
- **문서 간 의존성 명시 없음**: 이 도메인 정의서를 기반으로 생성될 후속 산출물(ERD, API 명세, 기능 명세서, 테스트 계획)과의 연관 관계가 정의되어 있지 않다. 도메인 정의가 변경될 때 어떤 하위 문서를 함께 업데이트해야 하는지 알 수 없다.
- **리뷰 및 승인 이력 없음**: 작성자, 검토자, 승인자 및 각 역할의 서명일 정보가 없다. 이해관계자 합의 없이 단방향으로 발행된 문서는 향후 요구사항 분쟁 시 공식 근거로 사용하기 어렵다.
- **섹션 간 상호 참조 링크 미흡**: 비즈니스 규칙에서 도메인 모델 속성을 언급할 때(예: BR-04의 title, BR-13의 dueDate) 해당 모델 섹션으로의 내부 앵커 링크가 없어 대형 문서로 확장 시 탐색 비용이 증가한다.

### 종합 총평

TodoListApp 도메인 정의서 v1.0.0은 개인 할일 관리 애플리케이션의 핵심 도메인을 식별하고 기술하는 데 있어 입문 수준을 명확히 상회하는 완성도를 보인다. 특히 범위와 범위 제외의 명시적 분리, 도메인별로 그룹화된 비즈니스 규칙, 그리고 일관된 용어 정의 테이블은 팀 내 공유 언어(Ubiquitous Language) 형성에 효과적으로 기여한다.

그러나 문서가 설계 및 개발 단계로 넘어가기 위한 실질적인 계약 문서로 기능하기에는 세 가지 영역에서 보완이 필요하다. 첫째, 비즈니스 규칙과 유스케이스 간의 명시적 연결이 없어 개발자가 규칙의 완전한 구현을 보장하기 어렵다. 둘째, 수락 기준(Acceptance Criteria)이 없어 테스트 단계에서 독립적인 테스트 케이스 작성이 불가능하다. 셋째, 비밀번호 정책, 필드 길이 제약 등 정량적 기준이 없어 구현 단계에서 임의 해석의 여지가 남아 있다.

종합 가중 점수 **3.5 / 5.0** 은 "개발 착수 조건부 승인" 수준으로 평가한다. 아래 우선순위 개선 권고사항을 반영한 후 v1.1.0으로 재발행할 것을 권고한다.

## 4. 우선순위 개선 권고사항 Top 3

---

### [P1] 유스케이스에 적용 비즈니스 규칙 명시적 참조 추가

**대상 기준**: 추적성 (Traceability)
**우선순위 근거**: 추적성 결함은 개발 단계에서 규칙 누락 구현으로 이어지는 가장 직접적인 리스크이다. 수정 비용이 문서 레벨에서 가장 낮다.

**현재 상태**

UC-04(할일 등록)는 비즈니스 규칙 BR-04, BR-05를 적용해야 하지만, 유스케이스 기술문에 해당 참조가 없다. 개발자는 전체 BR 목록을 수동으로 검토하거나 경험에 의존하여 규칙을 구현해야 한다.

**권고 조치**

각 유스케이스에 "적용 규칙" 항목을 추가하여 해당 유스케이스 실행 시 적용되어야 할 BR 번호를 명시한다.

```
### UC-04: 할일 등록
- 행위자: 인증된 사용자
- 적용 규칙: BR-03, BR-04, BR-05
- 흐름: 제목, 설명, 종료예정일, 카테고리 입력 -> 유효성 검증 -> 할일 저장
- 전제조건: 로그인 상태, 카테고리 선택 가능 상태
```

**기대 효과**: 개발자 구현 누락 리스크 감소, QA 테스트 케이스 도출 시간 단축, 코드 리뷰 기준 명확화.

---

### [P2] 핵심 필드에 대한 유효성 제약 조건 및 수락 기준 정의

**대상 기준**: 검증 가능성 (Verifiability), 완전성 (Completeness)
**우선순위 근거**: 정량적 기준 부재는 개발자, QA, 디자이너가 각각 다른 기준을 구현하게 만드는 설계 단편화(Design Fragmentation)의 원인이다. API 명세 작성 전에 반드시 확정되어야 한다.

**현재 상태**

비밀번호 복잡도, title 최대 길이, 카테고리 이름 최대 길이에 대한 명시적 기준이 없다. "유효성 검증" 단계가 여러 유스케이스에 등장하지만 검증 기준이 문서 어디에도 없다.

**권고 조치**

섹션 4 또는 별도 섹션으로 "필드 유효성 규칙" 테이블을 추가한다.

| 엔티티 | 필드 | 제약 조건 |
|--------|------|-----------|
| User | email | RFC 5321 형식, 최대 255자, 고유값 |
| User | password | 최소 8자, 영문+숫자 조합 필수 |
| User | name | 최소 1자, 최대 50자 |
| Todo | title | 최소 1자, 최대 200자 |
| Todo | description | 최대 1000자 (선택) |
| Category | name | 최소 1자, 최대 50자 |

아울러 주요 유스케이스(UC-01, UC-04, UC-08)에 Given-When-Then 형식의 수락 기준 예시를 1~2개씩 추가한다.

**기대 효과**: 경계값 테스트 케이스 도출 가능, DB 스키마 설계 기준 확정, UI 입력 제어 기준 통일.

---

### [P3] 변경 이력 테이블 및 BR-UC 매핑 매트릭스 추가

**대상 기준**: 유지보수성 (Maintainability), 추적성 (Traceability)
**우선순위 근거**: 프로젝트 진행에 따라 요구사항은 반드시 변한다. 변경 이력과 매핑 매트릭스는 문서를 단순 초기 산출물이 아닌 살아있는 계약 문서로 만드는 기반 구조이다.

**현재 상태**

버전 1.0.0으로 표기되어 있으나 변경 이력이 없고, 비즈니스 규칙과 유스케이스의 관계를 한눈에 파악할 수 있는 매핑이 없다.
