## 1. 문서 개요

본 문서는 인증 기반 개인 할일 관리 애플리케이션의 제품 요구사항을 정의한다. 개발팀이 MVP를 3일 내 구현할 수 있도록 기능 요구사항, 비기능 요구사항, 기술 아키텍처, UI/UX 방향을 명시한다.

---

## 2. 제품 목표

### 2.1 문제 정의

20대~50대 직장인은 업무와 개인 생활에서 발생하는 다수의 할일을 체계적으로 관리할 수단이 부족하다. 기존 메모 앱이나 노트는 카테고리 분류 및 완료 추적이 미흡하여 할일 누락과 우선순위 혼란이 발생한다.

### 2.2 제품 비전

사용자가 자신의 계정으로 안전하게 로그인하여 할일을 카테고리별로 등록, 조회, 완료 처리할 수 있는 경량 개인 생산성 도구를 제공한다. 복잡한 협업 기능 없이 단순하고 직관적인 개인 워크플로우 관리에 집중한다.

### 2.3 성공 지표 (KPI)

| KPI | 목표 기준 |
|-----|----------|
| 회원가입 완료율 | 가입 시도 대비 완료 80% 이상 |
| 할일 등록 후 완료 처리율 | 등록된 할일의 60% 이상 완료 상태 전환 |
| 카테고리 활용률 | 활성 사용자의 50% 이상 사용자 정의 카테고리 1개 이상 생성 |
| 세션 유지율 | 최초 로그인 후 7일 내 재방문율 40% 이상 |
| 시스템 가용성 | 99% 이상 (소규모 프로젝트 기준) |

## 3. 사용자 페르소나

### 페르소나: 김지현 (32세, 마케팅 팀장)

| 항목 | 내용 |
|------|------|
| 연령/직군 | 30대 초반, 직장인 |
| 디지털 숙련도 | 중상 (스마트폰, 웹 서비스 일상 활용) |
| 주요 사용 환경 | 사무실 PC + 이동 중 모바일 웹 |
| 할일 특성 | 업무 프로젝트, 개인 약속, 운동 루틴 등 카테고리 혼재 |

**주요 Pain Point**

- 업무 할일과 개인 할일이 섞여 우선순위 파악이 어렵다.
- 완료한 항목을 기록으로 남겨 진행 상황을 확인하고 싶다.
- PC에서 등록한 할일을 이동 중 모바일에서도 동일하게 확인하고 싶다.
- 마감일이 있는 항목을 기간 기준으로 빠르게 필터링하고 싶다.

---

#### UC-01: 회원가입

**설명:** 비회원이 이메일, 비밀번호, 이름을 입력하여 계정을 생성한다.

**사전 조건:** 미인증 상태

**기본 흐름:**
1. 사용자가 이메일, 비밀번호, 이름을 입력하고 가입 요청을 제출한다.
2. 시스템이 이메일 중복 여부를 검증한다.
3. 비밀번호를 암호화하여 저장한다.
4. 계정 생성 완료 후 로그인 화면으로 이동한다.

**예외 흐름:**
- 이메일 중복 시 오류 메시지를 반환한다.
- 필수 입력값 누락 시 필드별 유효성 오류를 표시한다.

**비즈니스 규칙:** BR-01, BR-02

**입력 유효성:**
- 이메일: RFC 5322 형식 준수 (필수)
- 비밀번호: 8자 이상, 영문+숫자 조합 필수
- 이름: 1자 이상 필수

**수락 기준 (AC):**
- AC-01-1: Given 미가입 이메일과 유효한 비밀번호/이름 입력 When 가입 요청 Then 201 Created 반환 및 로그인 화면 이동
- AC-01-2: Given 이미 가입된 이메일 입력 When 가입 요청 Then 409 Conflict 반환 및 오류 메시지 표시
- AC-01-3: Given 비밀번호 7자 이하 입력 When 가입 요청 Then 400 Bad Request 반환 및 필드 오류 표시

---

#### UC-02: 로그인 / 로그아웃

**설명:** 등록된 사용자가 자격증명을 통해 인증 토큰을 발급받는다.

**기본 흐름:**
1. 사용자가 이메일과 비밀번호를 입력한다.
2. 시스템이 자격증명을 검증한다.
3. JWT 액세스 토큰을 발급하여 클라이언트에 반환한다.
4. 로그아웃 시 클라이언트 측 토큰을 삭제한다.

**예외 흐름:**
- 이메일 미존재 또는 비밀번호 불일치 시 단일 오류 메시지 반환 (보안상 구체적 원인 미노출).

**비즈니스 규칙:** BR-03

**기술 참고:** JWT 기반 인증, 향후 OAuth 확장 대비 인증 레이어 추상화 권장

**수락 기준 (AC):**
- AC-02-1: Given 올바른 이메일/비밀번호 입력 When 로그인 요청 Then 200 OK + JWT 액세스 토큰 반환
- AC-02-2: Given 잘못된 비밀번호 입력 When 로그인 요청 Then 401 Unauthorized 반환, 구체적 원인 미노출
- AC-02-3: Given 로그인 상태 When 로그아웃 요청 Then 클라이언트 토큰 삭제 및 로그인 화면 이동

---

#### UC-03: 개인정보 수정

**설명:** 인증된 사용자가 이름 또는 비밀번호를 변경한다.

**기본 흐름:**
1. 사용자가 변경할 항목(이름 또는 비밀번호)을 입력한다.
2. 비밀번호 변경 시 현재 비밀번호를 확인한다.
3. 변경 사항을 저장한다.

**비즈니스 규칙:** BR-02 (비밀번호 암호화 재적용)

---

#### UC-04: 할일 등록

**설명:** 인증된 사용자가 할일 항목을 생성한다.

**기본 흐름:**
1. 사용자가 제목(필수), 설명(선택), 종료예정일(선택), 카테고리(필수)를 입력한다.
2. 시스템이 입력값을 검증한다.
3. 할일을 저장하고 목록에 반영한다.

**예외 흐름:**
- 제목 누락 시 오류 반환.
- 카테고리 미선택 시 오류 반환.

**비즈니스 규칙:** BR-03, BR-04, BR-05

**기본값:**
- isCompleted: false
- dueDate: null (선택 입력)

**수락 기준 (AC):**
- AC-04-1: Given 인증된 사용자, 유효한 제목+카테고리 입력 When 등록 요청 Then 201 Created + 생성된 할일 반환
- AC-04-2: Given 제목 빈값 When 등록 요청 Then 400 Bad Request 반환
- AC-04-3: Given 카테고리 미선택 When 등록 요청 Then 400 Bad Request 반환

---

#### UC-05: 할일 목록 조회 및 필터링

**설명:** 인증된 사용자가 자신의 할일 목록을 조회하고 필터링한다.

**기본 흐름:**
1. 인증 토큰으로 본인 소유 할일만 조회한다.
2. 필터 조건(카테고리, 기간, 완료 여부) 단독 또는 복합 적용이 가능하다.
3. 필터 미적용 시 전체 할일을 반환한다.

**필터 상세:**
| 필터 | 기준 필드 | 비고 |
|------|----------|------|
| 카테고리 | categoryId | 단일 선택 |
| 기간 | dueDate | 시작일~종료일 범위 (이상/이하 포함) |
| 완료 여부 | isCompleted | true / false |

복합 필터 적용 시 조건 간 AND 연산으로 처리한다.

**비즈니스 규칙:** BR-06, BR-07, BR-12, BR-13

**수락 기준 (AC):**
- AC-05-1: Given 인증된 사용자 When 필터 없이 조회 Then 본인 소유 전체 할일 반환
- AC-05-2: Given 카테고리+완료여부 복합 필터 When 조회 Then 두 조건을 AND로 충족하는 할일만 반환
- AC-05-3: Given 타 사용자의 할일 ID When 직접 접근 Then 403 Forbidden 반환

---

#### UC-06: 할일 수정 및 완료 처리

**설명:** 인증된 사용자가 본인 할일의 내용을 수정하거나 완료 상태를 변경한다.

**기본 흐름:**
1. 사용자가 수정할 필드(제목, 설명, 종료예정일, 카테고리, 완료 여부)를 변경한다.
2. 시스템이 소유권을 확인한다.
3. 변경 사항을 저장하고 updatedAt을 갱신한다.

**완료 여부 토글:** isCompleted는 완료(true) ↔ 미완료(false) 양방향 전환이 가능하다. 완료 처리 후에도 미완료로 되돌릴 수 있다.

**비즈니스 규칙:** BR-04, BR-05, BR-06

**수락 기준 (AC):**
- AC-06-1: Given 미완료 할일 When 완료 처리 Then isCompleted=true 저장 및 updatedAt 갱신
- AC-06-2: Given 완료된 할일 When 미완료로 전환 Then isCompleted=false 저장 및 updatedAt 갱신
- AC-06-3: Given 타 사용자 소유 할일 When 수정 요청 Then 403 Forbidden 반환

---

#### UC-07: 할일 삭제

**설명:** 인증된 사용자가 본인 할일을 삭제한다.

**기본 흐름:**
1. 사용자가 삭제 대상 할일을 선택한다.
2. 시스템이 소유권을 확인한다.
3. 할일을 영구 삭제한다.

**비즈니스 규칙:** BR-06, BR-07

**수락 기준 (AC):**
- AC-07-1: Given 본인 소유 할일 When 삭제 요청 Then 200 OK + DB에서 영구 삭제 확인
- AC-07-2: Given 타 사용자 소유 할일 When 삭제 요청 Then 403 Forbidden 반환

---

#### UC-08: 사용자 정의 카테고리 관리

**설명:** 인증된 사용자가 커스텀 카테고리를 생성, 수정, 삭제한다.

**생성 흐름:**
1. 사용자가 카테고리 이름을 입력한다.
2. 동일 사용자 범위 내 이름 중복을 검증한다.
3. 카테고리를 저장한다.

**수정 흐름:**
1. 사용자가 본인이 생성한 카테고리 이름을 변경한다.
2. 기본 카테고리 여부를 확인하여 수정을 차단한다.

**삭제 흐름:**
1. 사용자가 삭제 대상 카테고리를 선택한다.
2. 해당 카테고리에 속한 할일이 있는지 검증한다.
3. 할일이 없는 경우에만 삭제를 허용한다.

**비즈니스 규칙:** BR-08, BR-09, BR-10, BR-11

**기본 카테고리 목록 (수정/삭제 불가, seed 데이터로 삽입):**
- 업무, 개인, 기타
- "전체"는 UI 필터 옵션이며 DB 카테고리 항목이 아니다.

**수락 기준 (AC):**
- AC-08-1: Given 신규 카테고리 이름 입력 When 생성 요청 Then 201 Created + 카테고리 반환
- AC-08-2: Given 동일 사용자가 이미 사용 중인 이름 입력 When 생성 요청 Then 409 Conflict 반환
- AC-08-3: Given 할일이 존재하는 카테고리 When 삭제 요청 Then 400 Bad Request 반환
- AC-08-4: Given 기본 카테고리 When 수정/삭제 요청 Then 403 Forbidden 반환

---

#### UC-09: 회원 탈퇴

**설명:** 인증된 사용자가 본인 계정을 삭제하고 모든 관련 데이터를 즉시 제거한다.

**사전 조건:** 로그인 상태

**기본 흐름:**
1. 사용자가 탈퇴를 요청한다.
2. 현재 비밀번호를 입력하여 본인 확인을 한다.
3. 시스템이 해당 사용자의 Todo, Category(Custom), User 데이터를 CASCADE DELETE로 일괄 삭제한다.
4. 클라이언트 토큰을 삭제하고 서비스 종료 화면으로 이동한다.

**예외 흐름:**
- 비밀번호 불일치 시 탈퇴 거부 및 401 반환.

**비즈니스 규칙:** BR-02, BR-03

**데이터 삭제 방식:** DB CASCADE DELETE 사용 (users 삭제 시 todos, custom categories 자동 삭제 보장)

**수락 기준 (AC):**
- AC-09-1: Given 올바른 비밀번호 확인 When 탈퇴 요청 Then 200 OK + users/todos/categories 전체 삭제 확인
- AC-09-2: Given 잘못된 비밀번호 입력 When 탈퇴 요청 Then 401 Unauthorized 반환, 데이터 보존
- AC-09-3: Given 탈퇴 완료 후 When 동일 계정으로 로그인 시도 Then 401 Unauthorized 반환

---

#### UC-10: 다국어 지원 (i18n) 및 언어 설정

**설명:** 사용자가 로그인한 후 선호하는 언어(한국어/영어)를 선택하여 UI 전체를 해당 언어로 변환한다.

**기본 흐름:**
1. 로그인 시 사용자의 저장된 언어 설정(`users.language`)을 조회한다.
2. 프론트엔드는 i18n 라이브러리(`react-i18next`)를 초기화할 때 사용자의 언어로 설정한다.
3. 사용자가 언어 설정 페이지에서 언어를 선택하면 `PATCH /api/users/me`로 `language` 필드를 업데이트한다.
4. 성공 시 프론트엔드는 즉시 UI 언어를 전환한다.

**지원 언어:**
- 한국어 (ko) — 기본값
- 영어 (en)

**비즈니스 규칙:** BR-14

**기본 카테고리명 다국어 처리:**
- DB에는 기본 카테고리 이름이 고정된 한국어 이름(업무, 개인, 기타)으로 저장된다.
- 프론트엔드는 카테고리 렌더링 시 `getCategoryDisplayName(category, t)` 유틸 함수를 사용하여 i18n 번역 키로 변환한다.
- 기본 카테고리: `category.defaultNames.work`, `category.defaultNames.personal`, `category.defaultNames.etc`

**수락 기준 (AC):**
- AC-10-1: Given 한국어 사용자 로그인 When 페이지 진입 Then 모든 UI가 한국어로 표시됨
- AC-10-2: Given 영어로 언어 변경 선택 When `PATCH /api/users/me` 호출 Then 200 OK + 모든 UI가 영어로 전환
- AC-10-3: Given 새로운 사용자 회원가입 When 기본값 ko 저장 Then 로그인 후 한국어 UI 표시
- AC-10-4: Given 기본 카테고리 렌더링 When 프론트엔드 언어 en 전환 Then 기본 카테고리 이름도 영어로 표시

## 6. 기술 아키텍처 개요

### 6.1 기술 스택

| 영역 | 기술 |
|------|------|
| 프론트엔드 | React 19+, TypeScript, Zustand (전역 상태), TanStack Query (서버 상태) |
| 백엔드 | Node.js, Express (REST API) |
| 데이터베이스 | PostgreSQL 17 |
| DB 연동 | pg 라이브러리 (ORM 사용 금지, Raw SQL) |
| 인증 | JWT (jsonwebtoken) |
| 비밀번호 암호화 | bcrypt |

### 6.2 시스템 구성도 (텍스트 기반)

```
[Client: Browser / Mobile Web]
        |
        | HTTPS REST API
        v
[Frontend: React 19 + TypeScript]
  - Zustand: 인증 상태, UI 상태 관리
  - TanStack Query: 서버 데이터 캐싱 및 동기화
        |
        | HTTP Request (JWT Bearer Token)
        v
[Backend: Node.js + Express]
  - Auth Middleware: JWT 검증
  - Router: /auth, /todos, /categories, /users
  - Controller -> Service -> Repository 레이어 구조
        |
        | pg (Raw SQL)
        v
[Database: PostgreSQL 17]
  - Tables: users, todos, categories
```

### 6.3 핵심 기술 결정 사항 및 근거

| 결정 사항 | 근거 |
|----------|------|
| ORM 사용 금지, pg 라이브러리 직접 사용 | SQL 제어권 확보, 불필요한 추상화 레이어 제거, 소규모 프로젝트에 적합 |
| Zustand 선택 | Redux 대비 보일러플레이트 최소화, 소규모 상태 관리에 적합 |
| TanStack Query 선택 | 서버 상태 캐싱, 로딩/오류 처리 일관화, 불필요한 API 재호출 방지 |
| JWT 인증 | Stateless 구조로 서버 세션 관리 불필요, 향후 OAuth 확장 시 토큰 기반 유지 가능 |
| PostgreSQL 17 | ACID 트랜잭션 보장, UUID 기본 지원, 관계형 데이터 모델에 최적 |

### 6.4 도메인 모델 (DB 테이블 요약)

**users**
```
id          UUID        PK
email       VARCHAR     UNIQUE NOT NULL
password    VARCHAR     NOT NULL  (bcrypt hash)
name        VARCHAR     NOT NULL
dark_mode   BOOLEAN     DEFAULT false
language    VARCHAR(10) NOT NULL DEFAULT 'ko'
created_at  TIMESTAMP   DEFAULT NOW()
updated_at  TIMESTAMP   DEFAULT NOW()
```

**categories**
```
id          UUID        PK
user_id     UUID        FK -> users(id) ON DELETE CASCADE, NULL = 기본 카테고리
name        VARCHAR     NOT NULL
is_default  BOOLEAN     DEFAULT false
created_at  TIMESTAMP   DEFAULT NOW()
updated_at  TIMESTAMP   DEFAULT NOW()
UNIQUE(user_id, name)
```

**todos**
```
id            UUID        PK
user_id       UUID        FK -> users(id) ON DELETE CASCADE NOT NULL
category_id   UUID        FK -> categories(id) NOT NULL
title         VARCHAR     NOT NULL
description   TEXT        NULLABLE
due_date      DATE        NULLABLE
is_completed  BOOLEAN     DEFAULT false
created_at    TIMESTAMP   DEFAULT NOW()
updated_at    TIMESTAMP   DEFAULT NOW()
```

| 항목 | 내용 |
|------|------|
| 인증 토큰 관리 | JWT Access Token을 Zustand authStore 메모리에 보관. localStorage/sessionStorage/Cookie 저장 금지. Refresh Token 미사용 (페이지 새로고침 시 재로그인) |
