## 1. 프로젝트 개요

### 목적

개인 사용자가 인증 기반으로 자신만의 할일을 등록하고 카테고리별로 체계적으로 관리할 수 있는 할일 관리 애플리케이션을 제공한다.

### 범위

- 사용자 인증 및 계정 관리 (회원가입, 로그인, 로그아웃, 개인정보 수정, 회원 탈퇴)
- 할일(Todo) 등록, 조회, 수정, 삭제
- 카테고리 관리 (기본 카테고리 제공 및 사용자 정의 카테고리 추가)
- 할일 목록 필터링 (카테고리별, 기간별, 완료 여부별)

### 범위 제외

- 팀 또는 다수 사용자 간 할일 공유 및 협업
- 알림(Push Notification) 기능
- 외부 캘린더 연동

## 2. 핵심 도메인 개념 (용어 정의)

| 용어 | 정의 |
|------|------|
| 사용자 (User) | 회원가입을 통해 시스템에 등록된 개인 계정 보유자 |
| 인증 (Authentication) | 사용자가 자신의 자격증명(이메일, 비밀번호)으로 신원을 확인하는 행위 |
| 할일 (Todo) | 인증된 사용자가 등록한 개별 작업 항목 |
| 카테고리 (Category) | 할일을 분류하기 위한 그룹 단위. 기본 카테고리와 사용자 정의 카테고리로 구분 |
| 기본 카테고리 (Default Category) | 시스템이 모든 사용자에게 공통으로 제공하는 카테고리. 확정 목록: 업무, 개인, 기타. 수정/삭제 불가 |
| 사용자 정의 카테고리 (Custom Category) | 인증된 사용자가 직접 생성한 카테고리. 생성자만 수정/삭제 가능 |
| 완료 여부 (Completion Status) | 할일의 처리 상태를 나타내는 값 (완료 / 미완료). 완료 ↔ 미완료 양방향 전환 가능 |
| 종료예정일 (Due Date) | 할일을 완료하기로 계획한 목표 날짜 |
| 회원 탈퇴 (Account Deletion) | 인증된 사용자가 본인 계정과 모든 관련 데이터를 즉시 영구 삭제하는 행위 |

## 3. 도메인 모델

### User (사용자)

| 속성 | 타입 | 설명 |
|------|------|------|
| id | UUID | 사용자 고유 식별자 |
| email | String | 로그인 및 식별에 사용되는 이메일 주소 (고유값) |
| password | String | 암호화된 비밀번호 |
| name | String | 사용자 표시 이름 |
| createdAt | DateTime | 계정 생성 일시 |
| updatedAt | DateTime | 계정 정보 최종 수정 일시 |

### Todo (할일)

| 속성 | 타입 | 설명 |
|------|------|------|
| id | UUID | 할일 고유 식별자 |
| userId | UUID | 할일 소유자 (User 참조) |
| categoryId | UUID | 분류 카테고리 (Category 참조) |
| title | String | 할일 제목 (필수) |
| description | String | 할일 상세 설명 (선택) |
| dueDate | Date | 종료예정일 (선택) |
| isCompleted | Boolean | 완료 여부 (기본값: false) |
| createdAt | DateTime | 할일 등록 일시 |
| updatedAt | DateTime | 할일 최종 수정 일시 |

### Category (카테고리)

| 속성 | 타입 | 설명 |
|------|------|------|
| id | UUID | 카테고리 고유 식별자 |
| userId | UUID | 소유자 (기본 카테고리의 경우 null, 사용자 탈퇴 시 CASCADE DELETE) |
| name | String | 카테고리 이름 |
| isDefault | Boolean | 시스템 기본 카테고리 여부 |
| createdAt | DateTime | 카테고리 생성 일시 |
| updatedAt | DateTime | 카테고리 최종 수정 일시 |

### 엔티티 관계

```
User ||--o{ Todo : "소유 (CASCADE DELETE)"
User ||--o{ Category : "생성 (Custom, CASCADE DELETE)"
Category ||--o{ Todo : "분류"
```

## 4. 핵심 비즈니스 규칙

### 사용자 및 인증

- BR-01: 이메일은 시스템 내 고유값이어야 하며, 중복 가입을 허용하지 않는다.
- BR-02: 비밀번호는 8자 이상, 영문+숫자 조합을 필수로 하며 반드시 bcrypt로 암호화하여 저장한다.
- BR-03: 인증되지 않은 사용자는 할일 및 카테고리에 접근할 수 없다.

### 할일 (Todo)

- BR-04: 할일의 제목(title)은 필수 입력값이다.
- BR-05: 할일은 반드시 특정 카테고리에 속해야 한다.
- BR-06: 사용자는 자신이 소유한 할일만 조회, 수정, 삭제할 수 있다.
- BR-07: 완료 처리된 할일도 삭제 전까지 목록에서 조회 가능하다.

### 카테고리 (Category)

- BR-08: 기본 카테고리는 모든 사용자가 공통으로 사용하며, 수정 및 삭제할 수 없다.
- BR-09: 사용자 정의 카테고리는 해당 카테고리를 생성한 사용자만 수정 및 삭제할 수 있다.
- BR-10: 카테고리 이름은 동일 사용자 범위 내에서 중복을 허용하지 않는다.
- BR-11: 할일이 존재하는 카테고리는 삭제할 수 없다.

### 필터링

- BR-12: 할일 목록 조회 시 카테고리, 기간(시작일~종료일), 완료 여부를 단독 또는 복합 조건으로 필터링할 수 있다. 복합 조건은 AND 연산으로 처리한다.
- BR-13: 기간 필터는 할일의 종료예정일(dueDate)을 기준으로 적용한다.

### 계정 관리

- BR-14: 회원 탈퇴 시 해당 사용자의 모든 데이터(Todo, Custom Category, User)를 즉시 영구 삭제한다. DB CASCADE DELETE로 무결성을 보장한다.

## 5. 주요 유스케이스

### UC-01: 회원가입

- 행위자: 비회원
- 흐름: 이메일, 비밀번호, 이름 입력 -> 이메일 중복 검증 -> 계정 생성
- 예외: 이메일 중복 시 가입 거부

### UC-02: 로그인 / 로그아웃

- 행위자: 가입된 사용자
- 흐름 (로그인): 이메일, 비밀번호 입력 -> 자격증명 검증 -> JWT 인증 토큰 발급
- 흐름 (로그아웃): 로그아웃 요청 -> 클라이언트 측 토큰 삭제
- 예외: 자격증명 불일치 시 로그인 거부 (구체적 원인 미노출)

### UC-03: 개인정보 수정

- 행위자: 인증된 사용자
- 흐름: 수정할 항목(이름, 비밀번호 등) 입력 -> 유효성 검증 -> 정보 업데이트
- 전제조건: 로그인 상태

### UC-04: 할일 등록

- 행위자: 인증된 사용자
- 흐름: 제목, 설명, 종료예정일, 카테고리 입력 -> 유효성 검증 -> 할일 저장
- 전제조건: 로그인 상태, 카테고리 선택 가능 상태

### UC-05: 할일 목록 조회 및 필터링

- 행위자: 인증된 사용자
- 흐름: 필터 조건 선택(카테고리 / 기간 / 완료 여부) -> 조건에 맞는 본인 소유 할일 목록 반환
- 전제조건: 로그인 상태

### UC-06: 할일 수정 및 완료 처리

- 행위자: 인증된 사용자
- 흐름: 할일 선택 -> 수정 항목 입력 또는 완료 여부 변경 -> 저장
- 전제조건: 로그인 상태, 본인 소유 할일
- 비고: isCompleted는 완료(true) ↔ 미완료(false) 양방향 전환 가능

### UC-07: 할일 삭제

- 행위자: 인증된 사용자
- 흐름: 할일 선택 -> 삭제 확인 -> 삭제 처리
- 전제조건: 로그인 상태, 본인 소유 할일

### UC-08: 사용자 정의 카테고리 관리

- 행위자: 인증된 사용자
- 흐름: 카테고리 이름 입력 -> 중복 검증 -> 생성 / 또는 기존 카테고리 선택 후 수정 및 삭제
- 전제조건: 로그인 상태
- 예외: 할일이 존재하는 카테고리 삭제 시 거부, 기본 카테고리 수정/삭제 시 거부

### UC-09: 회원 탈퇴

- 행위자: 인증된 사용자
- 흐름: 탈퇴 요청 -> 비밀번호 입력으로 본인 확인 -> 전체 데이터 CASCADE DELETE -> 토큰 삭제
- 전제조건: 로그인 상태
- 예외: 비밀번호 불일치 시 탈퇴 거부
