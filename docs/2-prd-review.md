## 1. 평가 개요

본 보고서는 TodoListApp의 도메인 정의서(이하 "문서 A")와 제품 요구사항 문서(이하 "문서 B / PRD")를 비교 분석하여 두 문서 간의 정합성을 평가한다. 평가 목적은 개발팀이 PRD를 기반으로 구현에 착수했을 때 도메인 정의서에서 합의된 비즈니스 규칙, 도메인 모델, 유스케이스가 올바르고 완전하게 반영되었는지 확인하는 것이다.

평가는 세 가지 관점으로 구성된다.

1. **정합성 검토 (Coverage Matrix)**: 문서 A의 항목별 PRD 반영 여부
2. **품질 평가**: 5개 기준 항목에 대한 5점 척도 점수 부여
3. **우선순위 개선 권고사항 Top 5**: 개발 착수 전 필수 조치 사항

## 2. 정합성 검토 (Coverage Matrix)

### 2.1 용어 정의 반영 현황

| 도메인 용어 (문서 A) | PRD 반영 여부 | 비고 |
|---------------------|--------------|------|
| 사용자 (User) | 반영 | UC 및 DB 스키마 전반에서 일관 사용 |
| 인증 (Authentication) | 반영 | JWT 기반으로 명확히 구체화됨 |
| 할일 (Todo) | 반영 | 기능 요구사항 및 DB 스키마에서 일관 사용 |
| 카테고리 (Category) | 반영 | 기능 요구사항 및 DB 스키마에서 일관 사용 |
| 기본 카테고리 (Default Category) | 반영 (부분) | PRD에서 "전체/업무/개인/기타"로 구체화됨. 문서 A의 예시(업무/개인/쇼핑)와 목록 불일치 — 협의 필요 표시는 있으나 미확정 상태 |
| 사용자 정의 카테고리 (Custom Category) | 반영 | UC-08 상세 기술에서 일관 사용 |
| 완료 여부 (Completion Status) | 반영 | isCompleted 필드명으로 구체화됨 |
| 종료예정일 (Due Date) | 반영 | dueDate 필드명으로 구체화됨 |

> **소결**: 용어 정의 8개 전체 반영. 단, 기본 카테고리 항목 예시가 두 문서 간 불일치 (문서 A: 업무/개인/쇼핑 vs PRD: 전체/업무/개인/기타).

### 3.2 항목별 평가 근거

---

#### 기준 1. 완전성 (Completeness) — 4.0 / 5

**잘된 점**

- 도메인 정의서의 BR-01 ~ BR-13 전체 13개 규칙이 PRD의 UC별 참조와 BR 요약 테이블에 빠짐없이 포함되었다. 단순 언급이 아니라 각 UC의 "비즈니스 규칙" 항목에 명시적으로 연결되어 있어 추적성이 확보되었다.
- UC-01 ~ UC-08 전체 유스케이스가 PRD의 기능 요구사항 섹션에 상세 기술되었으며, 도메인 정의서의 흐름 및 예외 조건보다 더 풍부한 정보(입력 유효성, 기본값, 기술 참고 사항)가 추가되었다.
- 비기능 요구사항(성능, 보안, 데이터 정책, 확장성)이 별도 섹션으로 구체적으로 기술되어 있어 도메인 정의서가 다루지 않은 운영 관점의 요구사항을 보완했다.
- 도메인 모델의 세 엔티티(User, Todo, Category) 및 관계가 PRD DB 스키마에 완전히 반영되었다.

**개선점**

- 회원 탈퇴(계정 삭제) 유스케이스가 두 문서 모두 부재하다. PRD 섹션 5.3에 "탈퇴 즉시 데이터 삭제" 정책이 명시되어 있으나, 이를 처리하는 UC(행위자, 흐름, 예외)가 정의되지 않았다. 개발팀이 구현 시 혼선이 발생할 수 있다.
- Category.updatedAt 속성이 도메인 모델과 DB 스키마 양쪽 모두에서 누락되어 있다. BR-09(사용자 정의 카테고리 수정 기능)가 존재함에도 변경 추적 필드가 없는 것은 도메인 완성도의 공백이다.
- BR-12의 복합 필터 조건에 대한 AND/OR 연산자 정의가 없어 구현 시 해석 차이가 생길 수 있다.

---

#### 기준 2. 명확성 (Clarity) — 4.0 / 5

**잘된 점**

- PRD의 UC별 상세 기술이 도메인 정의서 대비 유의미하게 구체화되었다. UC-01에 "이메일: RFC 5322 형식 준수", "비밀번호: 8자 이상 영문+숫자 조합 권장" 등 개발자가 즉시 적용 가능한 기준이 추가되었다.
- BR-03에 "401 반환", BR-06에 "403 반환"이 명시되어 오류 응답 처리의 모호성이 감소되었다.
- UC-08의 카테고리 관리가 생성/수정/삭제 흐름으로 분리 기술되어 하나의 UC 내 복합 동작이 각각의 흐름으로 명확히 구분된다.
- 기술 스택 선택 결정에 근거가 함께 제시되어(섹션 6.3) 개발팀의 구현 방향이 명확하다.

**개선점**

- 비밀번호 정책이 "8자 이상, 영문+숫자 조합 권장"으로 기술되어 "권장"과 "필수" 사이의 경계가 불명확하다. 구현자에 따라 숫자 미포함 8자 비밀번호를 허용할 수도 있다.
- UC-08 기본 카테고리 목록이 "전체, 업무, 개인, 기타 (실제 목록은 개발팀 협의 후 확정)"으로 기술되어 여전히 미확정 상태다. MVP 개발 Day 1에 DB seed 데이터를 작성해야 하는데 이 항목이 미결이면 Day 1 작업이 지연될 수 있다.
- JWT 리프레시 토큰 채택 여부가 "1차 선택 사항"으로 모호하게 기술되어 있다. 보안 요구사항과 직결되는 사항인데 개발팀이 자의적으로 결정하도록 위임된 상태다.
- UC-06에서 완료 처리 후 미완료로 되돌리는 토글 동작의 허용 여부가 기술되어 있지 않다. UI에는 완료/미완료 토글이 표시되어 있으나(섹션 7.2) 비즈니스 로직 레벨에서 이를 허용한다는 명시가 없다.

---

#### 기준 3. 추적성 (Traceability) — 4.5 / 5

**잘된 점**

- PRD는 도메인 정의서에서 제기된 가장 큰 추적성 결함인 "BR-UC 간 연결 없음" 문제를 해결했다. 각 UC 상세 기술에 "비즈니스 규칙: BR-XX, BR-YY" 형태로 명시적 참조가 포함되어 있어 개발자와 QA가 어떤 규칙을 검증해야 하는지 즉시 확인할 수 있다.
- UC-01 ~ UC-08의 MVP 우선순위 테이블(Must/Should/Could Have)이 유스케이스와 기능 항목을 연결하고 있어 범위 추적이 용이하다.
- DB 스키마의 제약 조건(NOT NULL, UNIQUE, FK, DEFAULT)이 비즈니스 규칙(BR-04, BR-05, BR-10 등)과 직접 대응되어 코드-규칙 간 추적성이 DB 수준까지 확장되었다.
- 출시 계획(Day 1/2/3)에 UC 번호가 명시되어 일정-기능 간 추적성이 확보되었다.

**개선점**

- BR-UC 매핑 매트릭스(전체 BR과 UC를 교차 표시한 표)가 없어 특정 BR이 변경될 때 영향받는 UC를 한눈에 파악하기 어렵다. (전 보고서에서 권고한 사항이 PRD에 미반영)
- UC-02에 로그아웃 기능이 추가되었으나, 로그아웃에 적용되는 비즈니스 규칙이 명시되지 않았다. 로그아웃은 BR-03(미인증 접근 제어)와 연관되므로 참조 추가가 필요하다.
- 탈퇴 데이터 삭제 정책(섹션 5.3)이 비즈니스 규칙 번호(BR-XX)로 관리되지 않아 후속 문서(API 명세, 테스트 계획)에서 참조할 식별자가 없다.

---

#### 기준 4. 검증 가능성 (Verifiability) — 3.0 / 5

**잘된 점**

- 성능 요구사항이 수치로 정의되어 있다(동시 접속 300명, API 응답 500ms 이내, 초기 로딩 3초 이내). 이는 성능 테스트의 합격 기준으로 활용 가능하다.
- KPI가 정량화되어 있어(가입완료율 80%, 완료처리율 60% 등) 출시 후 성공 여부 측정이 가능하다.
- UC-01의 입력 유효성 기준(이메일: RFC 5322, 비밀번호: 8자 이상)이 구체적으로 기술되어 해당 기준을 기반으로 경계값 테스트 케이스 도출이 가능하다.
- BR-03(401)과 BR-06(403)의 HTTP 상태코드 명시로 API 통합 테스트의 기대값이 명확해졌다.

**개선점**

- 수락 기준(Acceptance Criteria)이 어떤 유스케이스에도 정의되어 있지 않다. Given-When-Then 또는 체크리스트 형식의 수락 기준 없이는 QA가 독립적으로 테스트를 수행하기 어렵다.
- 비밀번호 정책 "영문+숫자 조합 권장"이 권장인지 필수인지 불명확하여 테스트 케이스 작성 시 기대값을 확정할 수 없다.
- 카테고리 이름, 할일 제목, 사용자 이름의 최대/최소 길이 제약이 없어 경계값 분석(Boundary Value Analysis) 테스트가 불가능하다. DB VARCHAR 컬럼의 실제 길이 제한이 명시되지 않아 오버플로우 테스트 기준이 없다.
- BR-12의 복합 필터 AND 조건 기대 결과에 대한 예시나 구체적 기술이 없어, 예를 들어 "카테고리=업무 AND 완료=false AND 기간=5월" 조건 적용 시 반환 결과를 사전에 검증할 수 없다.
- 시스템 가용성 99%가 KPI로 제시되어 있으나, 측정 방법(모니터링 도구, 측정 주기)이 없어 실제 검증 가능성이 낮다.

---

#### 기준 5. 일관성 (Consistency) — 3.5 / 5

**잘된 점**

- 도메인 용어(Todo, Category, User, isCompleted, dueDate 등)가 두 문서 전반에서 일관되게 사용되어 용어 혼용으로 인한 혼란이 없다.
- 비즈니스 규칙 번호 체계(BR-XX)가 두 문서 간 동일하게 유지되어 상호 참조가 자연스럽다.
- 엔티티 속성 명칭이 문서 A의 camelCase 기준과 PRD DB 스키마의 snake_case 기준으로 일관되게 변환되어 있어 의도적 설계 결정임이 명확하다.

**개선점**

- 기본 카테고리 예시 불일치: 문서 A에서는 "업무, 개인, 쇼핑"으로 예시를 제시하였고 PRD에서는 "전체, 업무, 개인, 기타"로 다르게 정의하였다. 두 문서 간 기준이 달라 어떤 것이 최종 확정값인지 불명확하다.
- UC-02의 범위 불일치: 도메인 정의서는 로그인만 정의하였으나 PRD는 로그인과 로그아웃을 하나의 UC로 묶었다. 이는 범위의 확장이지만 문서 A가 갱신되지 않아 불일치가 발생한다.
- 문서 A에서 "완료 여부(Completion Status)"를 용어로 정의하면서 값을 "완료/미완료"로 기술하였으나, PRD에서는 isCompleted: true/false의 Boolean 필드로만 다루어 비즈니스 용어와 기술 구현 간의 명시적 매핑이 없다.
- PRD 섹션 9 가정 항목에 "탈퇴: CASCADE DELETE 또는 앱 레벨 일괄삭제"라고 기술되어 두 방식 중 하나를 선택하도록 위임하고 있다. 이는 두 방식이 데이터 무결성 처리 방식에서 실질적으로 다를 수 있으므로 구현 전 확정이 필요하다.

### 4.2 종합 총평

PRD(v1.0)는 도메인 정의서(v1.0.0)의 핵심 산출물을 충실하게 반영하였다. BR-01 ~ BR-13 전체와 UC-01 ~ UC-08 전체가 누락 없이 PRD에 포함되었고, 도메인 정의서의 검토 보고서(1-domain-definition-review.md)에서 지적된 가장 큰 결함인 "BR-UC 간 명시적 연결 없음"을 UC별 비즈니스 규칙 참조 항목을 통해 효과적으로 해소하였다. 또한 비기능 요구사항과 기술 아키텍처를 통해 도메인 정의서가 다루지 않은 구현 수준의 결정사항을 명확하게 추가한 점은 높이 평가된다.

그러나 두 가지 관점에서 보완이 필요하다.

첫째, **검증 가능성**의 부족이다. PRD의 요구사항이 개발 가능한 수준으로는 기술되어 있으나, QA가 독립적으로 테스트 케이스를 도출하기 위한 수락 기준(Acceptance Criteria)이 어느 UC에도 존재하지 않는다. MVP를 3일 내에 개발하는 일정이라면 QA와 개발이 동시에 진행되어야 할 가능성이 높으므로, 적어도 Must Have UC들에 대한 핵심 수락 기준이 필요하다.

둘째, **미결 사항이 개발 Day 1을 직접적으로 저해**한다. 기본 카테고리 목록(C-01)과 탈퇴 처리 방식(C-08)이 미확정 상태로 남아 있는데, 두 사항 모두 Day 1 DB 스키마 설계 및 seed 데이터 작성 시 결정이 필요한 항목이다. 이 상태로 개발에 착수하면 재작업 리스크가 발생한다.

종합 가중 점수 **3.85 / 5.0**은 "조건부 승인" 수준으로 평가한다. 아래 P1~P2의 미결 사항을 개발 착수 전(Day 0)에 해소하고, P3~P5는 Day 1~2 내에 보완하여 PRD v1.1로 재발행할 것을 권고한다.

### [P1] 기본 카테고리 최종 목록 확정 — 개발 착수 전 필수

**대상 문서**: PRD 섹션 4.2 (UC-08), 섹션 9 (가정), 도메인 정의서 섹션 2
**관련 불일치**: C-01

**현재 문제**

문서 A는 "업무, 개인, 쇼핑"을 예시로 제시하고 있으며 PRD는 "전체, 업무, 개인, 기타 (협의 필요)"로 다르게 기술하고 있다. 두 문서 모두 확정값이 아닌 예시 또는 미결 상태로 남아 있다. Day 1 작업에 DB seed 데이터 작성이 포함되어 있으므로 이 항목이 미결이면 Day 1 작업이 부분적으로 불가능하다.

**권고 조치**

개발 착수 전 제품 책임자(PM)와 개발팀이 다음 결정을 확정하고 두 문서를 동시 업데이트한다.

1. 기본 카테고리 명칭 및 개수 확정 (예: 업무, 개인, 기타 — "전체"는 카테고리가 아닌 UI 필터 옵션이므로 카테고리 테이블에서 제외 검토 권고)
2. PRD UC-08의 "(협의 필요)" 표시를 확정값으로 교체
3. 도메인 정의서 섹션 2 기본 카테고리 예시를 확정값으로 업데이트

**기대 효과**: Day 1 DB seed 작성 즉시 착수 가능, 프론트엔드 카테고리 선택 UI 목록 확정.

### [P2] 회원 탈퇴 유스케이스(UC-09) 정의 추가

**대상 문서**: 도메인 정의서 섹션 5, PRD 섹션 4.2 및 섹션 5.3
**관련 불일치**: C-08

**현재 문제**

PRD 섹션 5.3에 "회원 탈퇴 즉시 해당 사용자의 모든 데이터(User, Todo, Category) 영구 삭제"라는 정책이 명시되어 있으며, 섹션 9 가정에 "CASCADE DELETE 또는 앱 레벨 일괄삭제" 두 가지 방식이 제시되어 있다. 그러나 이를 처리하는 UC가 없으며 처리 방식도 미결이다. /users 라우터가 기술 아키텍처에 포함되어 있으므로 탈퇴 API는 구현 대상임이 분명하다.

**권고 조치**

1. PRD 섹션 4.2에 UC-09 회원 탈퇴를 추가한다.

```
UC-09: 회원 탈퇴
행위자: 인증된 사용자
기본 흐름:
  1. 사용자가 탈퇴를 요청한다.
  2. 시스템이 현재 비밀번호를 재확인한다.
  3. 사용자의 모든 Todo, Custom Category, User 데이터를 삭제한다.
  4. 인증 토큰을 무효화한다.
비즈니스 규칙: BR-03 (인증 필요)
데이터 정책: 영구 삭제, 복구 불가
```

2. 섹션 9 가정에서 탈퇴 처리 방식을 확정한다. CASCADE DELETE를 권고 (DB 레벨 무결성 보장, 앱 레벨 오류 시에도 데이터 삭제 보장).
3. 도메인 정의서에 UC-09를 추가하고 관련 비즈니스 규칙(데이터 삭제 정책)을 BR-14로 추가한다.

**기대 효과**: /users DELETE API 구현 기준 명확화, QA 테스트 케이스 도출 가능, 데이터 삭제 정책 확정.

### [P3] 핵심 유스케이스 수락 기준(Acceptance Criteria) 추가

**대상 문서**: PRD 섹션 4.2
**관련 평가**: 검증 가능성 3.0점

**현재 문제**

UC별 상세 기술이 "흐름" 중심으로만 작성되어 있어 QA가 독립적으로 테스트 케이스를 도출하기 어렵다. MVP 일정(3일)을 고려하면 개발과 테스트가 병행되어야 하므로 수락 기준의 부재는 직접적인 품질 리스크다.

**권고 조치**

Must Have UC(UC-01, UC-02, UC-04, UC-05, UC-06, UC-07, UC-08) 각각에 최소 3개의 수락 기준을 Given-When-Then 형식으로 추가한다. 예시:

```
UC-01 수락 기준 예시:
- Given 미인증 사용자가 유효한 이메일(RFC 5322), 8자 이상 비밀번호, 이름을 입력할 때
  When 회원가입을 요청하면
  Then 201 응답과 함께 계정이 생성된다.

- Given 이미 가입된 이메일로 회원가입을 시도할 때
  When 가입 요청을 제출하면
  Then 409 응답과 이메일 중복 오류 메시지가 반환된다.

- Given 비밀번호가 7자인 경우
  When 가입 요청을 제출하면
  Then 400 응답과 비밀번호 유효성 오류 메시지가 반환된다.
```

**기대 효과**: QA 독립적 테스트 케이스 작성 가능, 개발자 구현 기준 명확화, UAT 기준 확보.

### [P4] Category.updatedAt 속성 추가 및 완료 상태 토글 정책 명시

**대상 문서**: 도메인 정의서 섹션 3, PRD 섹션 6.4 및 섹션 4.2 (UC-06)
**관련 불일치**: C-03, C-05

**현재 문제 1 — Category.updatedAt 누락**

BR-09에 의해 사용자 정의 카테고리 수정이 가능하나 Category 엔티티와 DB 스키마 모두에 updated_at이 없다. 수정 이력 추적이 불가능하며 이는 도메인 모델과 비즈니스 규칙 간의 내부 불일치다.

**권고 조치 1**

도메인 정의서 Category 모델에 `updatedAt | DateTime | 카테고리 최종 수정 일시` 속성을 추가한다. PRD DB 스키마 categories 테이블에 `updated_at TIMESTAMP DEFAULT NOW()` 컬럼을 추가한다.

**현재 문제 2 — 완료 상태 토글 정책 미정의**

UC-06은 "완료 여부 변경"을 기술하나, 완료 후 미완료로의 되돌리기(toggle)를 허용하는지 여부가 명시되지 않았다. PRD 섹션 7.2의 UI 화면에는 "완료/미완료 토글"이 언급되어 있어 기능 구현 의도가 있음은 알 수 있으나 비즈니스 로직 레벨에서의 확인이 없다.

**권고 조치 2**

PRD UC-06 기본 흐름에 "isCompleted는 true ↔ false 양방향 변경이 가능하다"를 명시한다. 도메인 정의서 BR 섹션에 해당 정책을 보조 규칙으로 추가하거나 UC-06 기술에 반영한다.

**기대 효과**: DB 스키마 일관성 확보, 프론트엔드 토글 UI 구현 기준 명확화.

### [P5] 비밀번호 정책 "권장" → "필수" 명확화 및 BR-UC 매핑 매트릭스 추가

**대상 문서**: PRD 섹션 4.2 (UC-01), PRD 섹션 4.3 (BR 요약)
**관련 평가**: 명확성 4.0점, 추적성 4.5점

**현재 문제 1 — 비밀번호 정책 모호성**

PRD UC-01에 "비밀번호: 8자 이상, 영문+숫자 조합 권장"으로 기술되어 있다. "권장"은 필수가 아닌 선택을 의미하나, 보안 요구사항 맥락에서는 필수로 처리해야 할 가능성이 높다. 개발자와 QA가 다르게 해석하면 구현 결과가 달라진다.

**권고 조치 1**

비밀번호 정책을 명확하게 결정하여 PRD를 다음 중 하나로 수정한다.
- 필수 적용: "비밀번호: 8자 이상, 영문+숫자 조합 필수"
- 권장 유지: "비밀번호: 8자 이상 필수, 영문+숫자 조합 권장 (미준수 시에도 가입 허용)"

**현재 문제 2 — BR-UC 매핑 매트릭스 부재**

BR과 UC 간 참조가 UC별 기술에는 있으나, 전체를 한눈에 확인할 수 있는 매핑 매트릭스가 없다. BR이 변경될 때 영향받는 UC를 즉시 식별하기 어렵다.
